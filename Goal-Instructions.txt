  You are taking over the Librarian repository:

  https://github.com/nampara-ai/librarian

  Librarian is a local-first document ingestion, cleaning, classification, and
  search system. It converts messy documents into Markdown/plain text, cleans
  them with an LLM while preserving fidelity, classifies them into Dewey
  Decimal-style categories, stores provenance-rich outputs, and exposes the same
  core engine through both CLI and FastAPI surfaces.

  The project began as a prototype agent that aggressively cleaned huge
  transcript corpora. The current repo is the production rewrite: hexagonal
  architecture, SQLite persistence, durable run records, queue workers,
  conversion/OCR adapters, versioned prompts, CLI/API parity, CI, CodeQL,
  Docker, and release automation.

  Current public release: `v0.1.0a6`.

  High-level code tour:

  - `src/librarian/domain/`
    Pure dataclasses and IDs. No framework/database/provider imports.

  - `src/librarian/application/`
    Core use cases and ports.
    Important files:
    - `ports.py`: protocol boundaries.
    - `ingest_document.py`: source ingestion and raw text persistence.
    - `convert_document.py`: single/batch conversion to Markdown/text.
    - `process_document.py`: full processing run lifecycle.
    - `clean_chunks.py`: fast/balanced/max-coherence chunk cleaning.
    - `assemble_document.py`: final output assembly and boundary cleanup.
    - `classify_document.py`: structured Dewey classification.
    - `jobs.py`: in-process runner and durable queue worker loop.
    - `import_library.py`: batch import workflow.

  - `src/librarian/ingest/`
    Extraction adapters:
    - text/json/csv/md
    - DOCX
    - PDF with page-aware extraction
    - image OCR via Tesseract
    - optional broad MarkItDown conversion
    PDF behavior is page-aware: embedded-text pages use embedded extraction;
  empty/scanned pages are OCRed. OCR defaults support up to 1,000 pages.

  - `src/librarian/storage/`
    SQLite implementation and migrations.
    SQLite is intentional and acceptable for alpha/local-first use. Do not
  propose replacing it just because it is SQLite. Focus instead on correctness,
  observability, scaling limits, and adapter boundaries.

  - `src/librarian/llm/`
    Providers:
    - mock provider for deterministic CI/local dry runs
    - OpenAI-compatible provider
    - lazy provider construction for ingest-only paths

  - `src/librarian/prompts/`
    Versioned prompts.
    Current defaults:
    - `cleaning/cmos_v2.md`
    - `classification/dewey_v2.md`
    Older v1 prompts remain for provenance/cache compatibility.

  - `src/librarian/cli/app.py`
    Typer CLI. Commands include init, migrate, convert, convert-dir, import,
  ingest, process, worker, search, export, benchmark, eval, api.

  - `src/librarian/api/app.py`
    FastAPI surface. Supports documents, imports, runs, events, search, config,
  metrics, health.

  - `tests/`
    Broad coverage for API, CLI, processing, SQLite, conversion, extractors,
  prompt loading, validation, queue behavior, eval/benchmark.

  - `docs/`
    Architecture, PRD, quickstart, conversion, deployment, release, supply

  You’re inheriting nampara-ai/librarian, a production rewrite of an old
  prototype agent called Librarian.

  Librarian is a local-first document/corpus cleaner and organizer. It ingests
  messy transcripts, PDFs, DOCX, Markdown, text-like files, and OCR images;
  converts them into Markdown/plain text; cleans them with an LLM while
  preserving source fidelity; classifies them into Dewey Decimal categories;
  persists runs and outputs in SQLite; and exposes the same workflow through
  both a CLI and FastAPI API.

  Recommendations For World-Class Production Readiness

  1. Build real long-document evaluation

  The biggest missing piece is not architecture; it is evidence. Create a
  sanitized benchmark/eval corpus with:

  - 500-1000 page embedded PDFs
  - scanned PDFs
  - mixed embedded/scanned PDFs
  - DOCX with tables, headers, footers, lists
  - OCR images
  - messy transcript files
  - multilingual OCR samples if needed

  Track:

  - conversion time
  - processing time
  - peak memory
  - output character ratio
  - page count integrity
  - search recall for known phrases
  - classification accuracy
  - OCR quality before/after LLM correction
  - formatting preservation

  Add a docs/results/ or release-attached benchmark artifact policy.

  2. Add page-level progress and resumability for huge PDFs

  Current PDF conversion works, but a true 1000-page scanned PDF should have
  durable page-level state.

  Recommended design:

  - Add page extraction records keyed by source hash + page number + extraction
    config.
  - Persist per-page status: pending, embedded, ocr, corrected, failed.
  - Allow resume after crash.
  - Stream page progress through CLI/API.
  - Sidecar should be written incrementally, not only after full conversion.
  - Failed pages should support retry.

  3. Improve OCR fidelity and diagnostics

  Current OCR path uses Tesseract and optional LLM correction. Next upgrades:

  - Capture Tesseract confidence via TSV/HOCR where available.
  - Implement LIBRARIAN_OCR_LLM_CORRECTION=low-confidence for real.
  - Preserve page images or OCR diagnostics optionally.
  - Add preprocessing knobs:
      - grayscale
      - thresholding
      - deskew
      - DPI tuning
  - Add structured page warnings in sidecars.
  - Add tests with actual scanned fixtures.

  4. Add real provider integration tests behind opt-in env vars

  CI should keep using mock, but maintainers should have opt-in tests:

  LIBRARIAN_RUN_PROVIDER_TESTS=1
  LIBRARIAN_LLM_PROVIDER=openai-compatible
  LIBRARIAN_LLM_MODEL=...

  Test:

  - cleaning prompt behavior
  - JSON classification response robustness
  - retry behavior
  - auth failure fast-fail
  - rate-limit retry
  - prompt version provenance
  - OCR correction quality

  5. Improve output rendering and formatting preservation

  The core goal is flawless Markdown/plain text. Add render-quality checks:

  - page markers preserved
  - Markdown headings stable
  - tables preserved where possible
  - lists preserved
  - citations not mangled
  - no leaked context markers
  - no assistant artifacts
  - no duplicated boundary sentences
  - no collapsed document into one paragraph

  Consider adding a Markdown AST validation layer or snapshot tests for
  representative fixtures.

  6. Make single-file import workflow smoother

  Right now librarian import is directory-oriented. Users naturally expect this
  to work:

  librarian import ./large.md --process

  Recommended fix:

  - Let import accept file or directory.
  - If file, convert/ingest/process just that file.
  - Keep convert-dir directory-only, but make import ergonomic.

  7. Strengthen API product surface

  API is functional, but production hosted use needs:

  - OpenAPI response models for all endpoints.
  - Better error taxonomy and stable machine-readable error codes.
  - Batch upload endpoint.
  - Import progress endpoints.
  - Run logs as structured JSON, not only events strings.
  - Auth model beyond one static API key if hosted.
  - Optional per-tenant data roots if multi-user.

  8. Add observability

  Current metrics are process-local and basic. Add:

  - structured logs for each run stage
  - run duration metrics
  - queue wait time
  - OCR pages/sec
  - LLM tokens/estimated cost if provider exposes usage
  - conversion failure counts by taxonomy
  - OpenTelemetry export option
  - Prometheus endpoint option

  9. Harden SQLite operations at scale

  SQLite is fine for this project, especially local-first. But tune and test it
  seriously:

  - Stress API + worker contention.
  - Validate WAL/busy timeout behavior under load.
  - Add VACUUM/checkpoint guidance.
  - Document expected DB growth for large corpora.
  - Consider filesystem/object content adapter later for very large cleaned/raw
    blobs, but don’t migrate away from SQLite prematurely.

  10. Improve search

  Current SQLite FTS is a good baseline. Next:

  - More forgiving query parser for punctuation/hyphenated terms.
  - Snippets/highlights.
  - Search over raw vs cleaned outputs.
  - Facets by classification/status/source.
  - Optional semantic index adapter behind a port.

  11. Release hardening

  Already good for alpha. Before beta:

  - Publish to PyPI if package ownership is settled.
  - Add artifact attestations.
  - Add signed releases.
  - Add container vulnerability scanning.
  - Consider locked constraints for release builds.
  - Add a reproducibility note.
  - Improve changelog release notes so release pages don’t include the full
    changelog as notes.

  12. Security/privacy pass

  Before hosted mode:

  - Threat model API imports.
  - Review symlink/path handling again.
  - Add malware/archive policy for uploaded files.
  - Add data deletion tests for every persisted artifact.
  - Ensure logs never include source text or API keys.
  - Add secret scanning guidance for users.
